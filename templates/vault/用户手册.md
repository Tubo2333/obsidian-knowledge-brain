# Obsidian 知识大脑 — 用户手册 / User Manual

> 给你（人类用户）的完整使用指南。AI 助手另有 [[README|README]] 入口。
> Complete guide for you (the human user). AI assistants have their own [[README|README]] entry point.

---

## 一、这是什么？/ 1. What Is This?

一句话：**把所有 AI 助手的"记忆"集中到一个 Obsidian Vault 里，让它们互相学习、不再遗忘。**

你每和一个 AI 聊完天，它会写一份会话纪要存到这里。每周日，Python 脚本自动扫描所有纪要，发现跨项目的错误模式，生成规则提案。你只需点点审批——规则就会自动同步到项目配置，所有 AI 都能读到。

Bottom line: **Consolidate all AI assistants' "memory" into one Obsidian Vault so they learn from each other and never forget.**

After each AI chat session, a summary is written here. Every Sunday, Python scripts auto-scan all summaries, discover cross-project error patterns, and generate rule proposals. You just click to approve — rules auto-sync to project configs for all AIs to read.

## 二、快速开始 / 2. Quick Start

### 2.1 初始化 / Initialize

```bash
cd {vault_path}/scripts/
python setup.py
```

脚本会问你 / The script will ask:
1. **Vault 路径 / Vault path** — Obsidian vault 放哪里？建议 `~/ObsidianBrain/`
2. **项目根目录 / Project root** — 你的代码项目根在哪？（里面有 CLAUDE.md 的那个目录）
3. **项目列表 / Project list** — 你要跟踪哪些项目？用逗号分隔，如 `web-app, data-pipeline`

初始化后你会得到 / After init you get:
- 完整的 vault 目录结构 / Full vault directory structure
- 所有模板文件 / All template files
- `config.yaml` 配置文件 / Config file
- `CLAUDE.md.patch` 快照（记录初始化时的状态）/ Snapshot of init state

### 2.2 配置 CLAUDE.md / Configure CLAUDE.md

把 `patches/CLAUDE.md.patch` 的内容粘贴到项目根目录的 `CLAUDE.md` 里。它添加了三段标记块：

```
<!-- COMPILED:RULES_START -->  ← compiler 自动填充规则
<!-- COMPILED:RULES_END -->

<!-- COMPILED:PROJECTS_START --> ← compiler 自动填充项目信息
<!-- COMPILED:PROJECTS_END -->

### Session Annotation Rules  ← 告诉 AI 怎么写 [DECISION]/[ERROR]/[SESSION_SUMMARY]
```

**不要手动编辑 COMPILED 块**——它们是 `compiler.py` 自动维护的。你在 vault 里审批完规则，周日脚本自动同步到这里。

**Do NOT manually edit COMPILED blocks** — they are auto-maintained by `compiler.py`. When you approve rules in the vault, Sunday's script auto-syncs them here.

### 2.3 第一次跑 / First Run

```bash
cd {vault_path}/scripts/
python runner.py --dry-run   # 先预览，不写文件 / Preview first
python runner.py             # 确认没问题后正式跑 / Run for real
```

### 2.4 配置定时任务 / Schedule Weekly Run

**Linux/macOS (cron):**
```cron
0 15 * * 0 cd {vault_path}/scripts && python runner.py
```

**Windows (Task Scheduler):**
```powershell
schtasks /create /tn "ObsidianBrain-WeeklyScan" /tr "python {vault_path}\scripts\runner.py" /sc weekly /d SUN /st 15:00
```

## 四、日常工作流 / 4. Daily Workflow

### 4.1 你和 AI 聊天时 / During Chat

你什么都不用做。AI 会自动：
- 在关键决策后写 `[DECISION: ...]`
- 遇到错误后写 `[ERROR: ...]`
- 会话结束时写 `[SESSION_SUMMARY]`

### 4.2 周日 / Sunday

脚本自动跑，你会得到：
- 一份**周报**（`04-Feedback/weekly-reports/{YYYY}-W{ww}.md`）
- 如果有跨项目模式（3+ 项目出现同一错误）→ 自动生成**审批卡**（`00-Rules/_inbox/` 里）

### 4.3 下次开 Claude / Next Chat Session

Claude 会话开始时，如果 `_inbox/` 里有待审批规则，AI 会提醒你：
```
⏳ 1 rule pending approval:
📌 redis_connection_timeout across 3 projects (5 occurrences)
[Y] 批准/Approve  [N] 拒绝/Reject  [M] 修改/Modify  [S] 跳过/Skip
```

### 4.4 审批流程 / Approval Flow

```
AI 提案 / AI proposes
  │
  ├─ [Y] 批准/Approve → 规则进入 beta（30天后自动转正）/ Rule enters beta (auto-promoted after 30 days)
  ├─ [N] 拒绝/Reject → 移到 _rejected/（30天后自动清理）/ Moved to _rejected/ (auto-cleaned after 30 days)
  ├─ [M] 修改/Modify → AI 按你的要求修改后重新提案 / AI revises and resubmits
  └─ [S] 跳过/Skip → 直到第 N 次触发再提醒 / Remind after N more triggers (see skip_until)
```

## 五、阅读周报 / 5. Reading Weekly Reports

周报在 `04-Feedback/weekly-reports/{YYYY}-W{ww}.md`。每周日自动生成。

周报包含 / Each report contains:
- **扫描统计 / Scan stats** — 扫了多少 session、花了多久
- **新增模式 / New patterns** — 这周发现了哪些新错误模式
- **规则审批 / Rule approvals** — 这周审批/归档了哪些规则
- **跨项目热点 / Cross-project hotspots** — 哪些错误跨了多个项目
- **元健康 / Meta health** — 扫描器本身有没有出问题

**看周报的最佳时机**：周日晚或周一早上，花 2 分钟过一遍。

## 六、手动触发扫描 / 6. Manual Scan Trigger

```bash
cd {vault_path}/scripts/
python runner.py              # 全流程 / Full pipeline
python runner.py --step analyze  # 只跑分析 / Analyze only
python runner.py --step report   # 只生成报告 / Report only
python runner.py --step maintain # 只跑审批卡 / Maintenance only
python runner.py --dry-run       # 预览，不写任何文件 / Preview without writing
```

## 七、回滚 / 7. Rollback

所有写操作都会先在 `04-Feedback/_rollback/` 里存一份备份。如果某次扫描搞坏了东西：

```bash
cd {vault_path}/scripts/
python runner.py --rollback {YYYY-MM-DD}   # 回滚到指定日期的状态 / Roll back to specified date
```

`_rollback/` 目录只保留最近 4 周的备份，超出的自动清理。

## 八、添加自定义错误分类 / 8. Adding Error Categories

编辑 `04-Feedback/error-taxonomy.md` 的 YAML frontmatter。模板里已经有一套通用分类：

```yaml
categories:
  - name: "R-plotting"
    # ...
  - name: "python-encoding"
    # ...
```

你可以：
- **删除**不需要的分类
- **添加**项目专属的分类（如 `k8s-deploy`、`sql-optimize`）
- **修改**子分类和正则匹配规则

分类的名字不重要（不会自动用于匹配），重要的是 `keywords` 字段——脚本用这些关键词从 session 里筛选候选。

## 九、常见问题 / 9. FAQ

### Q: 会不会很占空间？
A: 全是 Markdown 文本文件。100 个 session 纪要大约 500 KB。JSONL 备份略大但会定期清理。完全不用担心。

### Q: 我同时用多个 AI 工具（Claude、GPT、Gemini）能行吗？
A: 行。只要它们都能读写 Markdown 文件，就都能参与这个系统。关键是一致遵守 `[DECISION]`/`[ERROR]`/`[SESSION_SUMMARY]` 的格式约定。

### Q: 我不懂 Python 能跑起来吗？
A: setup.py 会引导你。只需要 Python 3.10+ 和 `pip install PyYAML`。定时任务不会配的话可以跳过，手动跑 `python runner.py` 也行。

### Q: 规则审批会不会烦死我？
A: 不会。审批卡只在两个条件下触发：(1) 同一个错误模式出现在 3+ 个项目中；(2) 你之前没有拒绝过类似的。而且你可以点 `S` 跳过——"再出现 N 次再提醒"。

### Q: AI 写的规则不靠谱怎么办？
A: 这是双向审批的意义——AI 提案、你决策。AI 不会自动应用任何规则。所有规则都要你点 Y 才生效。

### Q: 我的 vault 能放 GitHub 吗？
A: 可以。建议先加 `.gitignore` 排除 `04-Feedback/_logs/`、`_attachments/` 里的大文件。配置文件里如果有 API key 记得放 `.env` 而不要提交。

### Q: LLM 聚类要钱吗？
A: 可选功能。不配 API key 也能跑（用关键词模式）。配了 API key 后语义聚类更准，但因为有本地预筛选，每轮调用量很小（几 KB token），费用极低。
